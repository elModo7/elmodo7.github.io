Project Inspiration

When I visit a restaurant and try a dish I really like, I often feel inspired to replicate it at home. This impulse isn’t limited to food—it’s a mindset I also apply to software. I see a feature or service and think, “I can do that, maybe even better.”

I felt this hobby project was worth sharing, especially given the interesting technology stack I used. In many ways, this project mirrors and, in some cases, improves upon the functionality of Nintendo’s online Pokémon storage service, Pokémon Home.

Even though I don’t play video games much anymore due to the demands of adult life, I had been brainstorming this project for months. During my trips, I worked on diagrams, schematics, and mockups for the system. I also assembled a list of features I wanted to implement, along with the hardware and network topology, technology stack, and resources for video documentation. It also served as a great excuse to brush up on my English and video editing skills, which had gotten rusty after a five-year hiatus.

Frontend & Backend Development

For the frontend, I used a custom-built library, GBAMEM, which tracks the running emulator, retrieves base pointers, and maps them to the game’s memory address. The library interacts with a cut-down version of my pokecrystal wrapper to extract game data in a more user-friendly format, allowing me to convert memory data into JSON objects for seamless Pokémon uploads and downloads.

I also integrated AutoHotkey to implement smooth GUI interactions, preventing flickering during rapid updates. The client communicates with the backend via a set of API functions for user registration, authentication, and Pokémon data management.

On the backend, the system uses Spring Boot with JWT authentication, Spring Data (JPA & Hibernate), and Spring AOP for logging and monitoring. The API is connected to a MySQL database for persistent storage, and I containerized the backend using Docker to improve scalability and maintainability. For load balancing and routing, I used Zalando’s Skipper, ensuring a robust, distributed architecture.

I also experimented with shadow traffic techniques to monitor mirrored traffic behavior across different versions of the backend services.

Backstory of the project

After a long workday, I often find that time is scarce. Between the gym, household tasks, and social activities, my spare time is limited. However, on occasion, I manage to carve out time for my hobbies. If I’m not too exhausted, I spend these moments researching, developing, or documenting personal projects.

This past Christmas, I began creating my own GameBoy games and packaging them into physical cartridges, thanks to invaluable resources from InsideGadgets. During this time, a wave of nostalgia hit me, and I thought about revisiting older Pokémon games that my girlfriend had never had the chance to experience.

Rather than dealing with cables (wireless connectivity was not a thing for the second generation of Pokémon games) or struggling with slowdowns from GameBoy Color Link emulation over a shared network, I wanted a wireless solution. Sharing save data and playing together remotely can often result in frustrating lag, especially when you’re not in the same room. I envisioned creating my own cloud-based storage system to extend the capabilities of the classic GameBoy Color.

To bring this idea to life, I developed a full deployment for the system. The setup includes a desktop frontend application that dynamically interacts with the running emulator and the game. It bridges this data with a backend API service. This backend service was a great opportunity for me to practice microservices architecture while applying best practices using the Java Spring Boot framework.

I used Spring Boot to build a REST API, integrated Spring Data to connect to a MySQL database, and implemented Spring Security with JWT authentication for endpoint security. Additionally, I explored Spring AOP for monitoring data flow, tested Prometheus alongside Spring Actuator for metrics and health monitoring, and used Metricat for visualizing these metrics. Even though the service is designed to support a small user base (5-10 concurrent users), I used Zalando’s Skipper for load balancing between two backend instances, while also implementing rate limiting for endpoints like user registration, which could be an attack vector.

Challenges and Future Enhancements

I initially considered integrating Jenkins for CI/CD, but my current self-managed in-micro cron jobs and scheduled routines proved more than enough for this project’s scope. In the future, I would like to explore integrating Jenkins, Ansible, or Puppet to streamline deployment on some other larger scale hobby projects.

Other technologies I evaluated but decided against using included WebSockets and RabbitMQ, as the expected frequency of updates to the Pokémon bank made a REST API more appropriate.

This project, while rooted in nostalgia and personal interest, turned into an excellent opportunity to explore new technologies and practice each and every deployment step.
